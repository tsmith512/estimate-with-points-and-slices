Scrum is a process that follows Agile development principles, focusing on quick iteration and delivery of completed, usable, valuable functionality on a regular basis.

So a new project comes in…

Clients have big wishlist items.

Clients may specifically call out big stuff (e.g. shopping carts, user managment) or we may be recommending big functionality (e.g. editorial workflows or access control).

Those things aren't actionable directly because they're huge.

Let's call that an epic.

Epic
A “big ticket/story” or large piece of functionality. Not actionable or easy to estimate because of its size, complexity, or numerous parts.

What to do with an epic?

Break it down. Into thin vertical slices.

What are thin vertical slices?

What are thin vertical slices?

Accuracy of estimates drop as complexity or size increases.

Like making a to-do list.

Break epics into individual, actionable pieces of work.

Humans don't usually estimate very well, and the accuracy of estimates drops off as the thing being estimated increases.

It's also hard to wrap your head around a big unspecific blob.

A single ticket should be a single, deliverable unit of functionality.

Framing them as stories

It's easier to think about what a user should now be able to do when the thing is done.

By focusing on the user's goals and the actions we want to empower, you'll be able to write thinner units of work and you won't prescribe technical solutions in your backlog which gives the team more freedom.

Let's call that a User Story

User Story
As a    I want to    so that   .

An identified audience or user persona wants to perform a single action to accomplish a goal.

Not a Scrum requirement — an optional extra.

Example A: Cyclist

As a cyclist, I want to stop the bike so that I don't get hurt.

See how we have a specific type of person identified, a single action, and a clear goal? And I didn't say anything about brakes or how they should work.

Three Gotchas with Stories

Personas

Should be an actual audience. Be skeptical of “As a user,” stories.

Actions

Avoid technical prescription. Be focused and narrow.

Goals

Don't add surprise functionality here → that's a new story.

"As a user" is a basic blueprint, but it can obscure who we're building something for. Is this something "user" wants or something "marketing" wants?

Keep actions and goals simple, avoid dictating how a story should be implemented or inventing new functionality as part of the goal.

Example B: Don't Do This

Bad Story
As a user,
I would like to create an account during check-out
so that I can see a dashboard with all my past orders.
  • No identified audience
  • Technical prescription — no one actually wants to make an account, that's a hurdle for stuff we give users later
  • The goal describes a feature surprise

Example C: This is better

Better Story
As a customer,
I would like to save my shipping information
so that I don't have to enter it for each order.
  • We have a specific kind of user
  • We've described what they actually want to do
  • The goal addresses why a user wants to save the info
  • We've avoided technical prescription — this can actually be accomplished without user accounts

Thin Slices

Break down your epics.
Bite-sized pieces of work.
Be user and goal focused.
Consider user stories.

What are thin vertical slices?

Top-to-bottom implementation of a deliverable feature.

  • Frontend and backend
  • Content model and template
  • Admin UI and frontend component

A slice of cake

not a layer of cake.

Frontend, backend, components, databases, module install, content modeling, design.
Everything a user would need to use the feature complete the story.

If a thin slice is a small unit of functionality, the vertical slice is everything needed for it to be usable.

Push Past the Pushback…

Everyone will agree that this sounds like a great idea.

Then people silo themselves and you end up with frontend components that aren't hooked up to anything. Or admin UI elements that don't produce actions. Or some integration layer that doesn't get used. Team effort: if a ticket isn't usable by its intended audience, it isn't deliverable or valuable.

…because there will be Pushback

Spoilers!

This is not the most efficient way for an engineer to build something if they have perfect knowledge.

It is less efficient for an individual to build this way. This process is about maximizing value delivered for money and time spent: by a whole team, so we can deliver features continuously, and while remaining adaptable to change.

And sometimes it means you have to be okay with some rework, going back to the same model and adding one thing at a time, ripping something out to rewrite, or asking someone to feel like they aren't being efficient. Everything in the sprint has been deemed more urgent than anything that isn't.

Frontend and Backend Tickets?

Please no.

Keep tickets user/goal/value focused.

Splitting tickets splits team, harder to track progress.

Multiple people can work on a single ticket.

And here's why

Split tickets? → Challenges of waterfall

Waterfall can work, but it's a very different way to work. And it is not adaptable to change, and it requires a serious investment in documentation.

It also means you spend a ton of money and time building stuff that doesn't work until you get the last piece in place.

If you do have an exception and need to split tickets out into horizontal layers, keep them together when possible, document heavily, and slice it as narrowly as possible.

Instead, we have tools for this

In Jira, consider extra board columns or subtasks

Columns help with workflows when every parent story has to go through the same processes, like QA

Subtasks: Splitting a ticket into steps or pieces. You can tackle 'em in any order and each can go into every status column. Jira keeps the parent issue open until all its subs resolve.

Thin Vertical Slices enable Iteration and Rapid Delivery

Keeps team in sync, working on the same areas.

Divides up big work into deliverable segments.

Reduces waste and cost as requirements change.

There are exceptions to every rule. On rare occasion, it can be useful to split tickets horizontally. Planning a content migration → content modeling. Big content rewrite → editorial tools.

But even then, you're building what a user needs, you're just looking at different users. But you have to make sure you account for every layer you haven't built yet for every feature you touched.

How do we estimate slices?

Hours? Story points!

Estimate Work by Complexity

Story Points
Estimate the complexity of a task compared to others, rather than the time someone thinks it would take to make.

Why not estimate hours?

Who's hours?

What's included?

Client expectations?

  • You write backend microservices. I write frontend components. We won't agree on the hours.
  • Do requirements gathering meetings, code reviews, deployments, testing, and rework get included in an hours estimate?
  • If a task is estimated at 3 hours, what if it doesn't take 3 hours? Do clients expect our estimates to be perfect? Will they start to expect an over or under estimation?

We can agree on complexity.

Add and theme a link field on a page. Easy peasy.

Add taxonomy-powered faceting to a Solr search. Uh oh.

I can't tell you how much a motorcycle or a car or an airplane weighs, but I can tell you that a motorcycle weighs less than both, and a car is somewhere in the middle.

And from there, we can extrapolate a timeline

Velocity

Over time, the number of story points a team can build in a sprint or given work cycle will tend toward an average.

It's less granular than hours, but usually more accurate. If you know the team allocation is stable, you can estimate the number of points they can complete each time.

In Scrum, teams should commit to a sprint they agree is realistic and then work to meet that commitment. That helps you build a schedule. Usually stakeholders are more interested in when a feature will be delivered than the specific hours spent on it anyway.

Informed Choices

Knowing a team's velocity and stories' sizes, the Product Owner can make an informed choice about what goes in a sprint.

Do you want four motorcycles and two cars, or do you want one airplane this sprint?

Don't share your point math

Velocity ÷ Hours per Cycle = Points per Hour

Just keep it to yourself.

Don't talk about that calculation. Maybe for forecasting, never written down. As soon as you talk about it, the team starts to pick story points based on the hours calculation and clients start to assume that points are a fixed hours multiplier.

So... how?

Planning Poker, Spreadsheet Reveal, Shouting

Whatever works for your team, but the team picks the size, not the PO/stakeholders.

  • ScrumMaster: Help by pushing the team to compare new stories against completed stories, not hours.
  • Product Owner: Help by asking if there are ways to make the requirement simpler to achieve.
  • Neither of these people can vote or dictate a size, with one exception: early in a project, if the team is locked up and unsure, just pick something straightforward, call it a 3 and use that as a reference. The team will self-correct over time.

With “Pseudo Fibonacci” Numbers

Or: A scale that accounts for uncertainty at size.

1 = Check one box.
It's never just a 1.

20 or 40? Epic.
Break it down.

Not pictured:
2, 40, and 100.

Encourage your team to pick a size by comparing the scope of other stories, not time estimates.

One reason we broke epics into slices is because humans have a hard time estimating as the thing gets bigger. Story points account for this by increasing the gap between values.

Shirt sizes can help if points are a problem. Small = 3. XL = 13. I don't like it as much, but you can only say "points are not hours" so many times. If you can't get away from hours, use shirt sizes.

Also, 20, 40, and 100 are valid estimates, but that's an epic to break down. They make great budget placeholder tickets.

Stick to the Sizes

Don't use numbers in the middle. Round up to account for complexity and unknowns. Round down to timebox.

Don't change size retroactively if easier or harder than anticipated. Trends will reflect in velocity.

And don't change the size of a story if it isn't finished at the end of a sprint. That story rolls into the next sprint. Yes, there will be a drop in points this sprint, but that reflects that less was deliverable and completed this sprint. And next sprint will get a boost for the quick finish, so it'll even out.

Also, when gathering information for a story or working on adjustments, the size is a litmus test for if something is a scope change. Is it still an 8?

How can our tools help?

Sample workflow and reports from Jira.

Subtasks

To split the “layers” of a ticket.

  • Subtasks track through sprints until the parent closes.
  • Assign different people, different statuses to each piece.
  • Each task has its own detail view and comment threads.

As product managers, we own the epics, tickets, and user stories. I propose that we don't own the subtasks. We should help engineers manage them in the issue tracker system. We can even make our own on occasion. But subtasks are for the team to document approach steps and horizontal layers.

This is where frontend vs. backend goes.

Columns

To add workflow steps

  • If each ticket needs the same steps in the same order.
  • Team members can remain assigned or hand-off.
  • Counterpoint: this can get messy if it gets too broad.
Columns read: To Do, Needs Info (!!!), In Backend, Ready for Frontend, In Frontend, Code Review, PO Review, Client Review, Done.
