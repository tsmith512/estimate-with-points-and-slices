4K uses Scrum…

…to manage our projects. I see estimating by complexity and building vertically to be the most useful parts of this system.

Scrum is a process that follows agile development principles, focusing on quick itertions and delivery of completed, usable functionality on a regular basis.

So a new project comes in…

Clients have big wishlist items.

Clients may specifically call out big stuff (e.g. shopping carts, user managment) or we may be recommending big functionality (e.g. editorial workflows or access control).

Those things aren't actionable directly because they're huge.

Let's call that an epic.

Epic
A “big ticket/story” or large piece of functionality. It's not actionable or easily estimated on its own because of its size or complexity, or because it has a lot of parts.
Depending on product lifecycle, an epic may represent a single unit of valueable functionality, so sometimes epics are only released at once, even if some parts are ready before others.

What to do with an epic?

Break it down. Into thin vertical slices.

What are thin vertical slices?

What are thin vertical slices?

Estimate precision drops as complexity/size increases.

Like making a to-do list.

Break epics into individual, actionable pieces of work.

Humans don't usually estimate very well, and the precision of estimates drops off as the size of the thing being estimated increases.

Beyond estimation, it's also harder to collaborate on a big undefined blob of work.

A single ticket should be a single, deliverable unit of functionality.

Framing them as Stories

It's easier to think about what a user should be able to do when the thing is complete.

By focusing on the user's desired actions and goals, you'll be able to write thinner units of work and you won't prescribe technical solutions in your backlog which gives the team more freedom.

Let's call that a User Story

User Story
As a    I want to    so that   .

An identified audience or user persona wants to perform a single action to accomplish a goal.

Not a Scrum requirement — an optional extra.

Example A: Cyclist

As a cyclist, I want to stop the bike so that I don't get hurt.

See how we have a specific type of user identified, a single action, and a clear goal?

Three Gotchas with Stories

Personas

Should be an actual audience. Be skeptical of “As a user,” stories.

Actions

Avoid technical prescription.

Goals

Don't add surprise functionality here → that's a new story.

"As a user" is a basic blueprint, but it can obscure who we're building something for. Is this something "user" wants or something "marketing" wants?

Keep actions and goals simple, avoid dictating how a story should be implemented or inventing new functionality as part of the goal.

Example B: Don't Do This

Bad Story
As a user,
I would like to create an account during check-out
so that I can see a dashboard with all my past orders.
  • No identified audience
  • Technical prescription — no one actually wants to make an account, that's a hurdle for stuff we give users later
  • Is check-out new?
  • The goal describes a feature surprise

Example C: This is better

Better Story
As a customer,
I would like to save my contact information
so that I don't have to enter it each time.

* Assumption: a rudimentary check-out process already exists.

  • We have a specific kind of user
  • We've described what they actually want to do
  • The goal addresses why a user wants to save the info
  • We've avoided technical prescription — this can actually be accomplished without user accounts

Thin Slices

Break down your epics.
Bite-sized pieces of work.
Be user and goal focused.
Consider user stories.

What are thin vertical slices?

Top-to-bottom implementation of a deliverable feature.

  • Frontend and backend
  • Content model and template
  • Admin UI and frontend component

A slice of cake

not a layer of cake.

Frontend, backend, components, databases, module install, content modeling, design.
Everything a user would need to use the feature complete the story.

If a thin slice is a small unit of functionality, the vertical slice is everything needed for it to be usable.

Push Past the Pushback…

Everyone will agree that this sounds like a great idea.

Then people silo themselves and you end up with frontend components that aren't hooked up to anything. Or admin UI elements that have no frontend effect. Or some integration layer that doesn't get used. Team effort: if a ticket isn't usable, it isn't deliverable or valuable.

…because there will be Pushback

Spoilers!

This is not the most efficient way for an individual to build something with perfect knowledge.

It is less efficient for an individual to build this way. This process is about maximizing value delivered for money and time spent while remaining adaptable to change.

And sometimes it means you have to be okay with some rework, throwing something away, or asking someone to feel like they aren't being efficient. Everything in the sprint has been deemed more urgent than anything that isn't in the sprint.

Frontend and Backend Tickets?

Please no.

Keep tickets user/value focused.

Splitting tickets splits team, harder to track progress.

Multiple people can work on a single ticket.

And here's why

Split tickets → Waterfall

Instead, we have tools for this

In Jira, consider extra board columns or subtasks

Column: self-explanatory, but imposes an "order" which may be unnecessary

Subtask: Jira will keep the parent issue open until every sub is closed, but you can tackle 'em in any order and each can go into every status column.

Thin Vertical Slices enables Iteration

Keeps team in sync, working on the same areas.

Divides up big work into deliverable segments.

Reduces waste and cost as requirements change.

There are exceptions to every rule. On rare occasion, it can be useful to split tickets horizontally. Planning a migration → content modeling. Big content rewrite → editorial tools.

But even then, you're building what a user needs, you're just looking at different users. But you have to make sure you account for every layer you haven't built yet for every feature you touched.

How do we estimate slices?

Hours? Story points!

Estimate Work by Complexity

Story Points
Estimate the relative complexity of a task rather than the time someone thinks it would take to complete.

Why not estimate hours?

Who's hours?

What's included?

Client expectations?

  • You write modules. I write CSS. We won't agree on the hours.
  • Do requirements gathering meetings, code reviews, deployments, testing, and rework get included in an hours estimate?
  • If a task is estimated at 3 hours, what if it doesn't take 3 hours? Do clients expect our estimates to be perfect? Will they start to expect an over or under estimation?

We can agree on complexity.

Add and theme a link field on a page. Easy peasy.

Add taxonomy-powered faceting to a Solr search. Uh oh.

I can't tell you how much a motorcycle or a car or an airplane weighs, but I can tell you that a motorcycle weigs less than both, and a car is somewhere in the middle.

And from there, we can extrapolate a timeline

Velocity

Over time, the number of story points a team can build in a sprint or given work cycle will tend toward an average.

It's less granular than hours, but often more accurate. If you know the team will work a set number of hours in a sprint/cycle, you can estimate the number of points they can complete.

In Scrum, teams should commit to a sprint they agree is realistic and then work to meet that commitment. That helps you build a schedule. Usually stakeholders are more interested in when a feature will be delivered.

Informed Choices

Knowing a team's velocity and stories' sizes, the Product Owner can make an informed choice about what goes in a sprint.

Do you want four motorcycles and two cars, or do you want one airplane this sprint?

Don't share your math

Velocity ÷ Hours per Cycle = Points per Hour

Just keep it to yourself.

Don't talk about that calculation. Use it for budget forecasting, but as soon as you talk about it, the team starts to pick story points based on the hours calculation and clients start to assume that points are a fixed hours multiplier.

So... how?

Planning Poker, Spreadsheet Reveal, Shouting

Whatever works for your team, but the team picks the size, not the PO/stakeholders.

The Scrum Master might help facilitate comparing a story to previous stories to help arrive at a size. The PO can ask if there's a way to trim complexity so that a story would be less complex to build. But a PO cannot dictate a size nor do they get a vote.

With “Pseudo Fibonacci” Numbers

Or: A scale that accounts for uncertainty at size.

1 = Check one box.
It's never just a 1.

20 or 40? Epic.
Break it down.

Not pictured:
2, 40, and 100.

Encourage your team to pick a size by comparing the scope of other stories, not time estimates.

One reason we broke epics into thin slices is because humans have a hard time estimating with precision as the thing becomes bigger or more complicated. This applies with story points, too.

Shirt sizes can help if points are a problem. Small = 3. XL = 13. I don't like it as much, but you can only say "points are not hours" so many times.

Also, 20 and 40 are valid estimates, but it's a sign that the team probably shouldn't take on the story until it is further defined and broken down.

Stick to the Sizes

Don't use numbers in the middle. Round up to account for complexity and unknowns. Round down to timebox.

Don't change size retroactively if easier or harder than anticipated. Trends will reflect in velocity.

And don't change the size of a story if it isn't finished at the end of a sprint. That story rolls into the next sprint. Yes, there will be a drop in points this sprint, but that reflects that less was deliverable and completed this sprint. And next sprint will get a boost for the quick finish, so it'll even out.

Also, when gathering information for a story or working on adjustments, the size is a litmus test for if something is a scope change. Is it still an 8?

How can our tools help?

Sample workflow and reports from Jira.

Subtasks

To split the “layers” of a ticket.

  • Subtasks track through sprints until the parent closes.
  • Assign different people, different statuses to each piece.
  • Each task has its own detail view and comment threads.

Columns

To add workflow steps

  • If each ticket needs the same steps in the same order.
  • Team members can remain assigned or hand-off.
  • This can get messy. Clearly... →
Columns read: To Do, Needs Info (!!!), In Backend, Ready for Frontend, In Frontend, Code Review, PO Review, Client Review, Done.

Velocity Chart

Version Report

Shows a timeline of: unsized, sized but incomplete, sized and complete issues.

Draws a line from 0 points at Release start date to completed points and current date.

Then extrapolates lines using current, optimistic, and pessimistic rate from current progress/date to all points completed.

Resulting in estimated date for release ready to ship.
